# Module 4 — Storage & Data: Volumes (Named)

Docker *volumes* are the preferred way to persist data generated by and used by containers. Unlike bind mounts, volumes are fully managed by Docker, stored in a part of the host filesystem, and survive container removal. They are ideal for databases and any stateful data.

<ExampleBox title="Create and use a named volume" lang="bash">
# Create a named volume
docker volume create app-data

# Mount it into a container at /var/lib/data
docker run -d -v app-data:/var/lib/data myapp

# List and inspect volumes
docker volume ls
docker volume inspect app-data
</ExampleBox>

Key points:
- `docker volume create` allocates a managed volume independent of any container.
- `-v app-data:/var/lib/data` mounts the volume at the container path; data written there persists.
- Volumes outlive their containers — removing the container does not delete the volume.
- Prefer volumes over writing inside the container's writable layer, which is ephemeral.

<ExerciseBox title="Persist data across containers" difficulty="Easy">
Create a named volume `app-data`, run a container that writes a file into the mounted path, then remove the container and start a new one with the same volume. Confirm the file is still present.
</ExerciseBox>

<ExerciseBox title="Inspect volume storage" difficulty="Medium">
Run `docker volume inspect app-data` to find the `Mountpoint` on the host. Use `docker volume ls` to list volumes and `docker system df -v` to see how much space named volumes consume. Explain the difference between a volume and the container's writable layer.
</ExerciseBox>
